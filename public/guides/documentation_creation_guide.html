Documentation Creation Guide

A comprehensive, topic-agnostic playbook for planning, writing, designing, reviewing, publishing, and maintaining documentation.

Core principle. Documentation is a product for readers. Its quality is measured by whether people can find, understand, trust, and apply it.

What this guide helps you create

Learning guides

Progressive instruction that builds understanding and skill.

How-to guides

Task-oriented steps for completing a specific goal.

Reference

Precise, consistently structured information for lookup.

Explanations

Mental models, rationale, relationships, and tradeoffs.

Processes and policies

Roles, workflow, requirements, exceptions, and accountability.

Operational runbooks

Detection, diagnosis, recovery, verification, and escalation.

The documentation lifecycle

discover→design→draft→validate→publish→maintain
End of sectionWelcome and purpose

Choose the documentation type

Most weak documentation mixes multiple reader goals into one page. Strong documentation separates learning, task completion, lookup, and explanation.

TypeReader questionTypical structure
TutorialCan you teach me?Guided sequence with a working result
How-toHow do I complete this task?Prerequisites, steps, verification
ReferenceWhat exactly is available?Consistent entries, tables, constraints
ExplanationWhy does this work this way?Concepts, models, rationale, tradeoffs
RunbookHow do I respond to this event?Detection, containment, diagnosis, recovery
PolicyWhat is required or prohibited?Scope, requirements, roles, exceptions

Example: one subject, four documents

Subject: backups. A tutorial teaches the first backup. A how-to restores a deleted project. A reference lists schedules and retention limits. An explanation describes incremental backups.

Research and verify the topic

Source hierarchy

  1. Authoritative sources: standards, policies, specifications, approved records.
  2. Observed behavior: tested product, workflow, or process.
  3. Subject-matter experts: people who perform, design, or approve the work.
  4. Existing documentation: useful evidence, but verify age and accuracy.
  5. Community sources: excellent for common questions and misunderstandings.

Research record

TEXT 
Claim or procedure:
Authoritative source:
Last verified:
Known exceptions:
Test performed:
Subject-matter reviewer:
Risk if incorrect:

Expert interview prompts

  • What do experienced people do automatically that beginners miss?
  • Where does the process usually fail?
  • Which steps are mandatory rather than customary?
  • What differs by role, version, or environment?
  • How can the reader verify success?

Design navigation and cross-links

Navigation layers

  • Global navigation: major content areas.
  • Local navigation: pages within the current area.
  • In-page navigation: headings for long pages.
  • Contextual links: prerequisites, next steps, and related topics.

Link-writing example

TEXT 
Avoid: Click here for more information.
Use: Review the account-recovery requirements.

Avoid: Learn more.
Use: Understand how retention periods are calculated.
Every page needs an exit. Give readers a next step, related reference, verification action, or escalation path.

Write useful titles and headings

Weak titleImproved title
ConfigurationConfigure weekly email reports
AccountsCreate and manage user accounts
ErrorsTroubleshoot failed data imports
BackgroundWhy approval requires two reviewers

Heading rules

  • Use one page title.
  • Keep headings parallel within a section.
  • Do not skip heading levels for appearance.
  • Front-load distinctive words.
  • Read only the headings to test whether the page still makes sense.
End of sectionWrite useful titles and headings

Write clearly and directly

Before and after

TEXT 
Before:
Upon successful completion of the authentication process,
the user will then be able to access the dashboard.

After:
After you sign in, the dashboard opens.

Writing rules

  • Put the main action near the beginning.
  • Use one main idea per sentence.
  • Prefer concrete nouns and strong verbs.
  • Use “you” when appropriate.
  • Keep conditions close to the action they modify.
  • Define unfamiliar terms at first use.
Plain language is not simplistic language. Preserve necessary precision while removing avoidable complexity.
End of sectionWrite clearly and directly

Manage terminology and definitions

Terminology table

MARKDOWN 
Preferred term | Definition | Avoid   | Notes
---------------|------------|---------|----------------
workspace      | ...        | project | Interface label
member         | ...        | user    | External docs
administrator  | ...        | admin   | Full term in policy

Rules

  • Use one preferred term for one concept.
  • Match interface labels exactly.
  • Avoid synonyms merely for variety.
  • Expand unfamiliar acronyms on first use.
  • Record deprecated, prohibited, and audience-specific terms.
End of sectionManage terminology and definitions

Write reliable procedures

Procedure anatomy

  1. Trigger or purpose
  2. Prerequisites
  3. Numbered actions
  4. Expected results
  5. Verification
  6. Recovery or escalation

Worked example

MARKDOWN 
# Enable weekly reports

## Before you begin
You must have administrator access.

## Steps
1. Open **Settings**.
2. Select **Notifications**.
3. Turn on **Weekly report**.
4. Select **Save**.

## Verify the result
A confirmation message appears and **Weekly report** displays **On**.

## If it does not work
Confirm that your account has administrator access.
Place warnings before the risky action. Readers act as they read.
End of sectionWrite reliable procedures

Use diagrams, screenshots, tables, and media

VisualBest use
FlowchartDecision paths and branching
Architecture diagramComponent relationships
Sequence diagramTiming and interaction order
ScreenshotLocation and visual recognition
TableComparison across consistent attributes
VideoMotion, timing, or physical technique

Visual guidance

  • Give every visual a clear question to answer.
  • Crop screenshots to the relevant area.
  • Remove personal or sensitive information.
  • Use annotations sparingly.
  • Do not rely on color alone.
  • Explain diagrams in nearby text.
  • Assign ownership because visuals age quickly.

Build accessible documentation

Accessibility checklist

  • Headings follow a meaningful hierarchy.
  • Links describe their destination.
  • Keyboard navigation works.
  • Focus indicators remain visible.
  • Contrast is sufficient.
  • Images have appropriate alternative text.
  • Color is not the only signal.
  • Tables have headers.
  • Layouts remain usable when text is enlarged.

Alternative text

Image purposeApproach
DecorativeUse empty alternative text
Functional iconDescribe the action, such as “Open settings”
Data chartState the main finding and provide underlying data
Process diagramSummarize the sequence and relationships in text
End of sectionBuild accessible documentation

YAML

      - lint-markdown
      - check-links

    const pages = [...document.querySelectorAll('.page')];

    let current = 0;

    function showPage(index) {
      current = Math.max(0, Math.min(index, pages.length - 1));
      pages.forEach((page, i) => page.classList.toggle('active', i === current));
      links.forEach((link, i) => link.classList.toggle('active', i === current));

Master checklists

Before drafting

  • Primary reader is defined.
  • Reader outcome is observable.
  • Scope is explicit.
  • Authoritative sources are identified.
  • The documentation type is appropriate.
  • Owner and reviewers are known.

Before publishing

  • Facts are verified.
  • Examples and procedures are tested.
  • Headings support scanning.
  • Warnings appear before risk.
  • Accessibility checks are complete.
  • Links work.
  • Owner and review date are present.

During maintenance

  • Changes trigger documentation review.
  • Stale pages are flagged.
  • Deprecated pages point to current guidance.
  • Search gaps and feedback are reviewed.
  • Examples still work.
