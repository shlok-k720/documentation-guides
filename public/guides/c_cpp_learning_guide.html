C and C++ for Experienced Programmers

A practical, documentation-style guide to the C language and modern C++. It assumes you already understand programming fundamentals and focuses on the concepts that are distinctive, subtle, or especially important in C-family systems programming.

Scope. The guide teaches C first, then builds modern C++ on top of it. Examples target C17 and C++20 where practical, while noting compatibility considerations.

Mental model

C gives you a thin, explicit abstraction over machine memory, data layout, and control flow. C++ keeps that power while adding stronger abstraction mechanisms, deterministic resource management, generic programming, and a broad standard library.

AreaCC++
Primary styleProcedural and data-orientedMulti-paradigm: procedural, object-oriented, generic, functional
Resource managementManual conventionsRAII and ownership types
StringsNull-terminated character arraysstd::string, views, and C strings
CollectionsArrays and custom structuresStandard containers and algorithms
Error handlingStatus codes, errnoExceptions, status types, optional values
CompilationTranslation units + linkerSame model, plus templates and modules

Recommended sequence

  1. Toolchain and compilation model. Understand what the compiler and linker actually do.
  2. C core. Types, pointers, arrays, functions, memory, structs, headers, and I/O.
  3. Reliability. Warnings, sanitizers, debugging, testing, and undefined behavior.
  4. Modern C++. RAII, classes, STL, templates, smart pointers, move semantics, and concurrency.
  5. Projects. Build small command-line tools, a reusable library, and one performance-sensitive program.

How compilation and linking work

C and C++ are usually compiled ahead of time. Each source file becomes an object file; the linker combines object files and libraries into an executable or shared library.

source→preprocessor→compiler→assembler→object file→linker→program

Translation units

A translation unit is roughly one source file after all included headers and preprocessor directives have been expanded. Names can be visible within one translation unit yet unavailable to another unless declared with appropriate linkage.

C 
// math_utils.h
#ifndef MATH_UTILS_H
#define MATH_UTILS_H

int square(int x);

#endif
C 
// math_utils.c
#include "math_utils.h"

int square(int x) {
    return x * x;
}
C 
// main.c
#include <stdio.h>
#include "math_utils.h"

int main(void) {
    printf("%d\n", square(12));
    return 0;
}
BASH 
gcc -std=c17 -Wall -Wextra -c math_utils.c -o math_utils.o
gcc -std=c17 -Wall -Wextra -c main.c -o main.o
gcc main.o math_utils.o -o demo

Declaration vs. definition

  • A declaration introduces a name and type to the compiler.
  • A definition creates the function body or allocates storage for an object.
  • Multiple compatible declarations are often allowed; multiple external definitions usually cause linker errors.

Common failure categories

StageTypical problemExample
PreprocessingMissing header or malformed macrofatal error: file.h not found
CompilationSyntax or type errorPassing char* where int is required
LinkingMissing or duplicate definitionundefined reference
RuntimeLogic, memory, or environment problemOut-of-bounds access

Headers, imports, and modules

Traditional C and C++ use textual inclusion with #include. C++20 also defines modules, which can reduce repeated parsing and improve dependency boundaries, but toolchain support and build integration vary. Learn headers first because they remain ubiquitous.

CPP 
#include <stdio.h>      // C standard header
#include <vector>       // C++ standard header
#include "my_api.hpp"   // project header
End of section How compilation and linking work

Program structure and conventions

The entry point

C 
#include <stdio.h>

int main(int argc, char *argv[]) {
    printf("Program: %s\n", argv[0]);

    for (int i = 1; i < argc; ++i) {
        printf("arg[%d] = %s\n", i, argv[i]);
    }

    return 0;
}

main returns an integer status to the environment. By convention, zero means success. Use EXIT_SUCCESS and EXIT_FAILURE from <stdlib.h> when that improves clarity.

Naming and formatting

The language does not prescribe one naming style. Consistency matters more than choosing snake_case versus camelCase. Distinguish types, constants, functions, and private members in a predictable way.

Namespaces in C++

CPP 
#include <iostream>
#include <string>

namespace telemetry {

struct Reading {
    std::string sensor;
    double value;
};

void print(const Reading& reading) {
    std::cout << reading.sensor << ": " << reading.value << '\n';
}

} // namespace telemetry

int main() {
    telemetry::Reading r{"temperature", 21.5};
    telemetry::print(r);
}
Avoid using namespace std; in headers. It injects many names into every file that includes the header and can create collisions.
End of section Program structure and conventions

Fundamental types

Boolean_Bool, boolC uses <stdbool.h> for bool; C++ has built-in bool.

Prefer fixed-width integers for binary formats

C 
#include <stdint.h>
#include <inttypes.h>
#include <stdio.h>

int main(void) {
    int32_t temperature_milli_c = 21500;
    uint64_t packet_id = UINT64_C(9000000000);

    printf("%" PRId32 "\n", temperature_milli_c);
    printf("%" PRIu64 "\n", packet_id);
}

Initialization

CPP 
int count = 0;          // copy initialization
double rate = 2.5;
const int limit = 100;  // read-only after initialization

// C++:
int retries{3};         // brace initialization
auto total = 0LL;       // deduced as long long
const auto name = std::string{"Ada"};

In C++, brace initialization rejects many narrowing conversions. auto asks the compiler to deduce the type from the initializer; it does not make the variable dynamically typed.

Integer promotions and signed/unsigned traps

C 
#include <stddef.h>

void example(size_t n) {
    for (size_t i = 0; i < n; ++i) {
        // correct: i and n have compatible unsigned types
    }

    // Avoid: int i compared with size_t n can produce warnings
}
Signed integer overflow is undefined behavior. Unsigned arithmetic wraps modulo 2N. Neither rule should be used accidentally.

Casts

C has one general cast syntax. C++ provides named casts that make intent reviewable:

  • static_cast<T> for checked compile-time conversions.
  • const_cast<T> for changing const qualification.
  • dynamic_cast<T> for runtime-checked polymorphic downcasts.
  • reinterpret_cast<T> for low-level representation conversions.
CPP 
double average = 9.7;
int truncated = static_cast<int>(average);

const unsigned char* bytes =
    reinterpret_cast<const unsigned char*>(&average);

Short-circuit evaluation

C 
if (ptr != NULL && ptr->ready) {
    // ptr->ready is evaluated only when ptr is non-null
}

Bit flags

C 
#include <stdint.h>
#include <stdbool.h>

enum Permission {
    PERM_READ  = 1u << 0,
    PERM_WRITE = 1u << 1,
    PERM_EXEC  = 1u << 2
};

bool can_write(uint32_t mask) {
    return (mask & PERM_WRITE) != 0;
}

Precedence is not a readability tool

Even when you know precedence rules, parenthesize mixed operations when the grouping is not obvious.

C 
int result = (a + b) * c;
bool in_range = (x >= low) && (x <= high);
Do not modify and read the same scalar object multiple times in an expression unless sequencing is guaranteed. Split complex expressions into named steps.

Conditionals, loops, and control flow

Branching

C 
if (score >= 90) {
    grade = 'A';
} else if (score >= 80) {
    grade = 'B';
} else {
    grade = 'C';
}

Switch

C 
enum State { STATE_IDLE, STATE_RUNNING, STATE_ERROR };

switch (state) {
case STATE_IDLE:
    start_work();
    break;
case STATE_RUNNING:
    poll();
    break;
case STATE_ERROR:
    recover();
    break;
default:
    /* defensive fallback */
    break;
}

Case labels fall through unless control exits. In C++17, use [[fallthrough]] to document intentional fallthrough.

Loops

C 
for (int i = 0; i < 10; ++i) {
    printf("%d\n", i);
}

while (has_more()) {
    process_next();
}

do {
    response = read_response();
} while (!valid(response));

Range-based loop in C++

CPP 
#include <vector>

std::vector<int> values{2, 4, 6};

for (int value : values) {
    // value is a copy
}

for (int& value : values) {
    value *= 2; // modifies the container
}

for (const int& value : values) {
    // read without copying
}

break, continue, and early return

Early return often reduces nesting. continue can clarify a loop when exceptional cases are handled first.

C 
int parse_positive(const char* text, int* out) {
    if (text == NULL || out == NULL) {
        return 0;
    }

    // parse...
    *out = 42;
    return 1;
}

Functions, parameters, scope, and linkage

Function declarations and calls

C 
double mean(const double values[], size_t count);

double mean(const double values[], size_t count) {
    double total = 0.0;

    for (size_t i = 0; i < count; ++i) {
        total += values[i];
    }

    return count == 0 ? 0.0 : total / count;
}

Array parameters decay to pointers. The function does not know the length unless you pass it separately or use a richer abstraction.

Pass-by-value, pointer, and reference

C 
// C: output parameter through pointer
int divide(int numerator, int denominator, int* result) {
    if (denominator == 0 || result == NULL) {
        return 0;
    }
    *result = numerator / denominator;
    return 1;
}
CPP 
// C++: reference parameters
void increment(int& value) {
    ++value;
}

double magnitude(const Vector& v) {
    // const reference avoids copying and prohibits mutation
    return /* ... */;
}

Function pointers and callbacks in C

C 
#include <stdlib.h>

int compare_ints(const void* lhs, const void* rhs) {
    const int a = *(const int*)lhs;
    const int b = *(const int*)rhs;
    return (a > b) - (a < b);
}

int main(void) {
    int values[] = {4, 1, 8, 2};
    qsort(values, 4, sizeof(values[0]), compare_ints);
}

Overloading and default arguments in C++

CPP 
int clamp(int value, int low, int high);
double clamp(double value, double low, double high);

void log(std::string_view message, int level = 1);

Scope and storage duration

  • Automatic: local variables usually exist until the block exits.
  • Static: objects exist for the entire program.
  • Dynamic: objects exist from allocation until explicit release or owner destruction.
  • Thread-local: one instance per thread.
C 
static int file_private_counter = 0; // internal linkage in C

void tick(void) {
    static int call_count = 0;       // static storage duration
    ++call_count;
    ++file_private_counter;
}

Arrays, C strings, and C++ strings

Fixed-size arrays

C 
int values[5] = {10, 20, 30, 40, 50};
size_t count = sizeof(values) / sizeof(values[0]);

for (size_t i = 0; i < count; ++i) {
    printf("%d\n", values[i]);
}
No automatic bounds checking. Accessing outside an array is undefined behavior, even if it appears to work.

C strings

A C string is a sequence of char terminated by a zero byte. The terminator consumes space and must be preserved.

C 
#include <stdio.h>
#include <string.h>

int main(void) {
    char name[32] = "Grace";
    size_t length = strlen(name);

    printf("%s has %zu characters\n", name, length);
}

Safe formatted output

C 
char buffer[64];
int written = snprintf(buffer, sizeof buffer, "id=%d", 42);

if (written < 0) {
    // encoding or formatting error
} else if ((size_t)written >= sizeof buffer) {
    // output was truncated
}

std::string and std::string_view

CPP 
#include <iostream>
#include <string>
#include <string_view>

bool starts_with(std::string_view text, std::string_view prefix) {
    return text.starts_with(prefix); // C++20
}

int main() {
    std::string message = "hello";
    message += " world";

    std::string_view view = message;
    std::cout << view << '\n';
}
Views do not own data. A std::string_view becomes invalid when the underlying character storage is destroyed or reallocated.

Multidimensional arrays

C 
enum { ROWS = 3, COLS = 4 };
int matrix[ROWS][COLS] = {0};

for (size_t r = 0; r < ROWS; ++r) {
    for (size_t c = 0; c < COLS; ++c) {
        matrix[r][c] = (int)(r * COLS + c);
    }
}
End of section Arrays, C strings, and C++ strings

Pointers, references, and const correctness

Pointer fundamentals

C 
int value = 42;
int* ptr = &value;

printf("%d\n", *ptr); // dereference
*ptr = 99;             // modifies value

A pointer stores an address. Dereferencing requires that the pointer be non-null, correctly aligned, and pointing to a live object of a compatible type.

Null pointers

CPP 
int* c_pointer = NULL;       // C
int* cpp_pointer = nullptr;  // C++

Pointer arithmetic

C 
int values[] = {10, 20, 30};
int* begin = values;
int* end = values + 3;

for (int* p = begin; p != end; ++p) {
    printf("%d\n", *p);
}

Pointer arithmetic is only defined within one array object and one-past its end. You may form a one-past pointer, but you may not dereference it.

Const combinations

C 
const int* p1;       // pointer to const int
int* const p2 = &x;  // const pointer to int
const int* const p3 = &x; // const pointer to const int

C++ references

CPP 
int score = 7;
int& alias = score;
const int& read_only = score;

alias = 9; // score becomes 9

A reference is an alias that must be initialized. In normal code it cannot be reseated. References still have lifetime requirements: a reference to a destroyed object dangles just like a pointer.

Prefer richer pointer abstractions in C++

  • Use references for required non-owning access.
  • Use raw pointers for nullable non-owning access.
  • Use std::unique_ptr or std::shared_ptr to express ownership.
  • Use std::span for contiguous ranges with a known size.
CPP 
#include <span>

int sum(std::span<const int> values) {
    int total = 0;
    for (int value : values) {
        total += value;
    }
    return total;
}
End of section Pointers, references, and const correctness

Structs, enums, unions, and data layout

Structures

C 
typedef struct {
    double x;
    double y;
} Point;

double squared_distance(Point a, Point b) {
    const double dx = a.x - b.x;
    const double dy = a.y - b.y;
    return dx * dx + dy * dy;
}

Enums

C 
enum Color {
    COLOR_RED,
    COLOR_GREEN,
    COLOR_BLUE
};
CPP 
enum class Color : unsigned char {
    red,
    green,
    blue
};

Color c = Color::green;

C++ scoped enums avoid implicit conversion to integers and keep enumerator names scoped.

Unions

C 
union Number {
    int as_int;
    float as_float;
};

A union overlays members in the same storage. Reading a member other than the active one is restricted and often non-portable. Prefer explicit serialization, memcpy for representation transfer, or std::variant in C++.

Tagged union in C

C 
enum ValueKind { VALUE_INT, VALUE_DOUBLE };

struct Value {
    enum ValueKind kind;
    union {
        int as_int;
        double as_double;
    } data;
};

Layout, padding, and alignment

C 
#include <stddef.h>
#include <stdio.h>

struct Record {
    char tag;
    int count;
    double value;
};

int main(void) {
    printf("size=%zu\n", sizeof(struct Record));
    printf("count offset=%zu\n", offsetof(struct Record, count));
}
Do not assume a struct's in-memory layout is a portable file or network format. Padding, endianness, widths, and alignment can differ.

C++ structured bindings

CPP 
#include <tuple>

auto [x, y, z] = std::tuple{1, 2.5, "three"};

Memory management and object lifetimes

Stack-like automatic storage

C 
void process(void) {
    int local = 0;      // lifetime ends at block exit
    int values[128];    // fixed-size automatic array
}

Dynamic allocation in C

C 
#include <stdlib.h>

size_t count = 100;
int* values = malloc(count * sizeof *values);

if (values == NULL) {
    /* allocation failed */
    return;
}

for (size_t i = 0; i < count; ++i) {
    values[i] = 0;
}

free(values);
values = NULL;

Resizable allocation

C 
size_t new_count = count * 2;
int* grown = realloc(values, new_count * sizeof *values);

if (grown == NULL) {
    free(values);  // original block still exists
    return;
}

values = grown;
count = new_count;

Common memory failures

Leak

Allocated memory is never released.

Use after free

A pointer is used after its object's lifetime ends.

Double free

The same allocation is released twice.

Buffer overflow

A write crosses the bounds of an object.

Dynamic objects in C++

CPP 
Widget* raw = new Widget{/*...*/};
delete raw;

Direct new/delete are legal but rarely the best ownership interface. Prefer automatic objects, containers, and smart pointers.

CPP 
#include <memory>
#include <vector>

auto widget = std::make_unique<Widget>();
std::vector<Widget> widgets;
widgets.emplace_back(/* constructor args */);

Lifetime rule of thumb

Every pointer, reference, iterator, span, and view is valid only while its referenced object remains alive and has not been invalidated by reallocation or structural modification.

Preprocessor, headers, macros, and conditional compilation

Header guards

C 
#ifndef PROJECT_BUFFER_H
#define PROJECT_BUFFER_H

#include <stddef.h>

struct Buffer {
    unsigned char* data;
    size_t size;
};

#endif

#pragma once is widely supported but not formally standardized. Header guards are fully portable.

Object-like and function-like macros

C 
#define ARRAY_COUNT(a) (sizeof(a) / sizeof((a)[0]))
#define MAX_RETRIES 5
Macros are token substitution, not functions. Arguments can be evaluated more than once, names ignore scope, and type checking happens only after expansion.
C 
#define BAD_SQUARE(x) ((x) * (x))

int i = 3;
int result = BAD_SQUARE(i++); // undefined behavior / multiple modification

Prefer language features

CPP 
// C
static inline int square(int x) {
    return x * x;
}

// C++
constexpr int square_cpp(int x) {
    return x * x;
}

Conditional compilation

C 
#if defined(_WIN32)
    #include <windows.h>
#elif defined(__APPLE__)
    #include <TargetConditionals.h>
#else
    #include <unistd.h>
#endif

Feature flags

Use compile-time flags for platform or build configuration, not as a substitute for normal runtime control flow.

C 
#ifdef ENABLE_TRACE
#define TRACE(msg) fprintf(stderr, "[trace] %s\n", (msg))
#else
#define TRACE(msg) ((void)0)
#endif

I/O, files, parsing, and C error handling

Standard streams

  • stdin: input
  • stdout: normal output
  • stderr: diagnostics
C 
#include <stdio.h>

fprintf(stdout, "result=%d\n", result);
fprintf(stderr, "error: invalid input\n");

File I/O

C 
#include <stdio.h>

int copy_file(const char* source_path, const char* dest_path) {
    FILE* in = fopen(source_path, "rb");
    if (in == NULL) {
        return 0;
    }

    FILE* out = fopen(dest_path, "wb");
    if (out == NULL) {
        fclose(in);
        return 0;
    }

    unsigned char buffer[4096];
    size_t n;
    int ok = 1;

    while ((n = fread(buffer, 1, sizeof buffer, in)) > 0) {
        if (fwrite(buffer, 1, n, out) != n) {
            ok = 0;
            break;
        }
    }

    if (ferror(in)) {
        ok = 0;
    }

    if (fclose(out) != 0) {
        ok = 0;
    }
    fclose(in);

    return ok;
}

Parsing integers robustly

C 
#include <errno.h>
#include <limits.h>
#include <stdlib.h>

int parse_int(const char* text, int* out) {
    char* end = NULL;
    errno = 0;
    long value = strtol(text, &end, 10);

    if (errno != 0 || end == text || *end != '\0') {
        return 0;
    }
    if (value < INT_MIN || value > INT_MAX) {
        return 0;
    }

    *out = (int)value;
    return 1;
}

Error conventions

C libraries commonly use return codes, sentinel values, output parameters, and errno. Document whether a function leaves resources open, whether partial output is valid, and who owns returned memory.

C++ stream example

CPP 
#include <fstream>
#include <stdexcept>
#include <string>

std::string read_all(const std::string& path) {
    std::ifstream input(path, std::ios::binary);
    if (!input) {
        throw std::runtime_error("cannot open " + path);
    }

    return {std::istreambuf_iterator<char>(input),
            std::istreambuf_iterator<char>()};
}

Debugging, testing, warnings, and sanitizers

Debugging workflow

  1. Reproduce the failure with the smallest stable input.
  2. Compile with warnings, debug symbols, and sanitizers.
  3. Inspect the first reported error, not only the final crash.
  4. Use a debugger to inspect state and control flow.
  5. Add a regression test before or alongside the fix.

C++ classes, constructors, destructors, and RAII

Encapsulation

CPP 
#include <stdexcept>
#include <string>

class BankAccount {
public:
    explicit BankAccount(std::string owner, double balance = 0.0)
        : owner_(std::move(owner)), balance_(balance) {
        if (balance < 0.0) {
            throw std::invalid_argument("negative initial balance");
        }
    }

    void deposit(double amount) {
        if (amount <= 0.0) {
            throw std::invalid_argument("deposit must be positive");
        }
        balance_ += amount;
    }

    [[nodiscard]] double balance() const noexcept {
        return balance_;
    }

private:
    std::string owner_;
    double balance_;
};

RAII

Resource Acquisition Is Initialization binds a resource to an object's lifetime. Construction establishes ownership; destruction releases it. Stack unwinding and early returns then become safe by default.

CPP 
#include <cstdio>
#include <stdexcept>

class File {
public:
    explicit File(const char* path, const char* mode)
        : handle_(std::fopen(path, mode)) {
        if (!handle_) {
            throw std::runtime_error("fopen failed");
        }
    }

    ~File() {
        std::fclose(handle_);
    }

    File(const File&) = delete;
    File& operator=(const File&) = delete;

    FILE* get() const noexcept { return handle_; }

private:
    FILE* handle_;
};

Rule of zero, three, and five

  • Rule of zero: prefer members that manage themselves, so your class needs no custom destructor, copy, or move operations.
  • Rule of three: if you define destructor, copy constructor, or copy assignment, you often need all three.
  • Rule of five: in modern C++, also consider move constructor and move assignment.

Inheritance and polymorphism

CPP 
struct Shape {
    virtual ~Shape() = default;
    virtual double area() const = 0;
};

struct Circle final : Shape {
    explicit Circle(double radius) : radius(radius) {}
    double area() const override { return 3.141592653589793 * radius * radius; }
    double radius;
};

Use inheritance for substitutable interfaces. Prefer composition when one object merely uses or contains another.

Smart pointers, value categories, and move semantics

Unique ownership

CPP 
#include <memory>

auto item = std::make_unique<Item>(42);
process(*item);

auto transferred = std::move(item);
// item is now empty; transferred owns the Item

Shared ownership

CPP 
auto shared = std::make_shared<Node>();
std::weak_ptr<Node> observer = shared;

if (auto locked = observer.lock()) {
    use(*locked);
}

Use shared_ptr only when ownership is genuinely shared. Cycles of shared owners leak; use weak_ptr for non-owning back-references.

Copy vs. move

CPP 
std::string make_message() {
    std::string result = "large result";
    return result; // usually elided; otherwise movable
}

std::string a = make_message();
std::string b = std::move(a);

std::move does not move anything by itself. It casts an expression to an rvalue category so a move operation may be selected. A moved-from object remains valid but its value is usually unspecified.

Perfect forwarding

CPP 
template <typename T, typename... Args>
std::unique_ptr<T> make_owned(Args&&... args) {
    return std::unique_ptr<T>(
        new T(std::forward<Args>(args)...)
    );
}

Throwing and catching

Exception safety levels

  • Strong guarantee: failure leaves state unchanged.

Capture modes

  • []: capture nothing.
  • [x]: capture x by value.
  • [&x]: capture x by reference.

Low-level topics: bytes, endianness, aliasing, and interop

Object representation

C 
#include <stdint.h>
#include <string.h>

uint32_t bits_of_float(float value) {
    uint32_t bits;
    memcpy(&bits, &value, sizeof bits);
    return bits;
}

memcpy is the portable C technique for moving object representations between same-sized storage. In C++20, std::bit_cast can express the same intent at compile time when constraints are met.

Endianness

Multi-byte numbers can be stored least-significant byte first or most-significant byte first. Network and file formats must define byte order explicitly.

C 
uint32_t read_u32_be(const unsigned char bytes[4]) {
    return ((uint32_t)bytes[0] << 24) |
           ((uint32_t)bytes[1] << 16) |
           ((uint32_t)bytes[2] << 8)  |
           ((uint32_t)bytes[3]);
}

Strict aliasing

The optimizer assumes most unrelated pointer types do not refer to the same object. Violating the language's aliasing rules can produce optimized builds that behave differently from debug builds.

volatile

volatile is for certain memory-mapped I/O and externally changing objects. It is not a thread-synchronization primitive and does not make operations atomic.

C and C++ interoperability

CPP

extern "C" {
#endif

int library_version(void);

#ifdef __cplusplus
}
#endif

extern "C" gives C linkage to declarations when included from C++. Keep cross-language interfaces simple: fixed-width scalars, explicit ownership, opaque handles, and stable calling conventions.

Performance, profiling, and data-oriented design

Measure before optimizing

  1. Define the metric: latency, throughput, memory, startup, binary size, or power.
  2. Build a repeatable benchmark with representative data.
  3. Profile to identify the actual hot path.
  4. Change one significant factor at a time.
  5. Verify correctness and remeasure.

Optimization levels

BASH 
# Debug
clang++ -O0 -g app.cpp -o app_debug

# Typical release
clang++ -O2 -DNDEBUG app.cpp -o app_release

# Aggressive; measure carefully
clang++ -O3 -DNDEBUG app.cpp -o app_fast

Data locality

CPP 
struct ParticleAoS {
    float x, y, z;
    float vx, vy, vz;
};

struct ParticlesSoA {
    std::vector<float> x, y, z;
    std::vector<float> vx, vy, vz;
};

Array-of-structures is natural for per-object work. Structure-of-arrays can improve cache behavior and vectorization when processing one field across many objects.

Allocation awareness

  • Reserve vector capacity when the approximate size is known.
  • Avoid repeated string concatenation in tight loops.
  • Prefer value semantics and contiguous storage.
  • Do not replace clear code with custom memory pools without evidence.
CPP 
std::vector<Record> records;
records.reserve(expected_count);

Complexity and constants

Asymptotic complexity matters, but cache misses, allocation, branching, and representation often dominate at real input sizes. Choose algorithms and data layout together.

Undefined behavior and portability

Undefined behavior means the language standard imposes no requirements on the result. Compilers may optimize under the assumption that undefined behavior never occurs.

Frequent sources

  • Out-of-bounds access
  • Dereferencing null, dangling, or misaligned pointers
  • Use of uninitialized values
  • Signed integer overflow
  • Invalid shifts
  • Data races
  • Multiple unsequenced modifications
  • Violating object lifetime or aliasing rules
  • Incorrect format specifiers
C 
int bad(int* values, size_t count) {
    return values[count]; // one past the final valid element
}

Implementation-defined and unspecified behavior

  • Implementation-defined: implementation chooses and documents a behavior, such as whether plain char is signed.
  • Unspecified: one of several valid behaviors may occur without documentation, such as some evaluation orders.

Portability checklist

  • Use fixed-width integers for serialized data.
  • Specify byte order.
  • Avoid assuming padding or alignment.
  • Compile on more than one compiler.
  • Run 32-bit or alternate-architecture tests when relevant.
  • Avoid platform APIs behind a narrow abstraction layer.
  • Keep warnings clean under GCC, Clang, and MSVC when possible.

-fsanitize=address,undefined

API design and coding practices

Design contracts explicitly

For every public function, define valid input, ownership, lifetime, thread safety, failure reporting, and side effects.

C interface example

C 
/*
 * Parses one record.
 *
 * input:  borrowed null-terminated string; must not be NULL.
 * output: required pointer; receives parsed value on success.
 * returns: 1 on success, 0 on invalid input.
 * thread safety: reentrant.
 */
int record_parse(const char* input, struct Record* output);

C++ interface example

CPP 
class Parser {
public:
    [[nodiscard]]
    std::optional<Record> parse(std::string_view input) const;
};

Core practices

  • Initialize objects immediately.
  • Keep ownership visible in types and names.
  • Keep functions focused and testable.
  • Prefer immutable state where practical.
  • Minimize global mutable data.
  • Use const aggressively but meaningfully.
  • Compile with strong warnings.
  • Prefer the rule of zero in C++.
  • Use standard library facilities before inventing replacements.
  • Document invariants, units, and boundary conditions.

What not to overdo

  • Do not create a class for every noun.
  • Do not template code that only ever needs one type.
  • Do not use inheritance when composition is clearer.
  • Do not expose raw owning pointers.
  • Do not compress logic into clever expressions that resist debugging.
