    Swift 6 for Java and Kotlin Programmers

    A tutorial-oriented guide to the Swift language, SwiftUI, and Apple development. You already know objects, types, collections, lambdas, and Android-style declarative UI. Here you will learn where Swift deliberately makes different tradeoffs.

    Version target: Swift 6. Examples use modern Swift concurrency and strict concurrency terminology. SwiftUI APIs evolve with Apple platforms, but the patterns here are intentionally durable.

    Your map from the JVM

    Java intuition

    Reference-oriented classes, nullable references, garbage collection, interfaces, checked exceptions.

    Kotlin intuition

    Null safety, data classes, extension functions, sealed types, coroutines, Compose-like state flow.

    Swift shift

    Value types by default, optionals, protocols, ARC, structured concurrency, and views described as values.

    Learning path

    language basics→types and ownership→concurrency→SwiftUI→Apple frameworks→advanced practice

    What you need

    • A Mac running a current Xcode release, or Swift installed from swift.org for command-line language practice.
    • Comfort reading Java and Kotlin. No Objective-C knowledge is required.
    • For iOS/macOS apps: an Apple ID; paid developer membership is only needed for certain device distribution and services.

    • Bundle: the signed app package, identified by a reverse-DNS bundle identifier.

    Variables, constants, and types

    Swift uses local type inference like Kotlin, but its let and var apply consistently to bindings, properties, parameters, and collection variables.

    Declarations and interpolation
    JAVA
    final String name = "Mina";
int score = 10;
score += 5;
System.out.println(name + ": " + score);
    KOTLIN
    val name = "Mina"
var score = 10
score += 5
println("$name: $score")
    SWIFT
    let name = "Mina"
var score = 10
score += 5
print("\(name): \(score)")

    Important differences

    ConceptSwiftJava/Kotlin comparison
    Constant bindingletfinal / val
    Mutable bindingvarordinary variable / var
    Type annotationlet count: Int = 3int count = 3 / val count: Int = 3
    IntegersInt is platform-word sizedUsually use int / Int; choose fixed-width Swift integers for a protocol or file format
    StringsUnicode-correct String, not random-access by integer offsetDo not assume Java UTF-16 indexing semantics
    Binding immutability is not deep immutability. A let class reference cannot be rebound, but its mutable object can still change. A let struct value cannot have its mutable properties changed.

    Optionals and null safety

    Swift's T? means a value is either .some(T) or .none. It occupies the same conceptual role as Kotlin T?, but has Swift-specific syntax and powerful pattern matching.

    Safely consume a possibly missing value
    JAVA
    String city = user.getCity();
if (city != null) {
    System.out.println(city.toUpperCase());
}
    KOTLIN
    val city: String? = user.city
city?.let { println(it.uppercase()) }
    SWIFT
    let city: String? = user.city
if let city {
    print(city.uppercased())
}

    Core operations

    SwiftMeaningKotlin analogy
    if let valueUnwrap only in this branchvalue?.let { ... }
    guard let value else { return }Early exit; unwrapped value remains available afterwardElvis + return, often more readable in Swift
    value ?? fallbackDefault if missingvalue ?: fallback
    try? operation()Convert a thrown error into an optionalrunCatching(...).getOrNull()
    value!Force unwrap; traps if absent!!
    Treat ! as a proof obligation. Use it only when an invariant makes absence impossible and that invariant is clear. Prefer guard let, default values, or a throwing initializer for ordinary input.

    Collections and strings

    Swift's standard collections are value types with copy-on-write storage. They feel Kotlin-like in syntax but have different APIs and performance properties.

    Transform then filter a collection
    JAVA
    var names = List.of("Ada", "Bob", "Ari");
var result = names.stream()
    .filter(n -> n.startsWith("A"))
    .map(String::toUpperCase)
    .toList();
    KOTLIN
    val names = listOf("Ada", "Bob", "Ari")
val result = names
    .filter { it.startsWith("A") }
    .map { it.uppercase() }
    SWIFT
    let names = ["Ada", "Bob", "Ari"]
let result = names
    .filter { $0.hasPrefix("A") }
    .map { $0.uppercased() }

    Collection reference

    Swift typeCreateNotes
    Array<Element> / [Element]var scores = [10, 20]Ordered, integer-indexed collection
    Dictionary<Key, Value> / [Key: Value]let ages = ["Ana": 14]Subscript lookup returns an optional
    Set<Element>let tags: Set = ["swift", "ui"]Elements must be Hashable
    Range0..<5, 0...5Half-open versus closed; use half-open for indexes

    Strings are not integer-indexed

    A Swift Character represents an extended grapheme cluster. Use text.startIndex, text.index(_:offsetBy:), or string APIs such as prefix, split, and firstIndex; do not translate Java's charAt(i) mechanically.

    Functions, closures, and errors

    Swift function calls emphasize readable argument labels. Closures are first-class values and become terse in collection pipelines and SwiftUI callbacks.

    A function with defaults and a closure callback
    JAVA
    static String greet(String name) {
    return "Hi, " + name;
}
names.forEach(name -> System.out.println(greet(name)));
    KOTLIN
    fun greet(name: String = "Guest") = "Hi, $name"
names.forEach { println(greet(it)) }
    SWIFT
    func greet(_ name: String = "Guest") -> String {
    "Hi, \(name)"
}
names.forEach { print(greet($0)) }

    External argument labels

    In func move(from start: Point, to end: Point), callers write move(from: a, to: b). The underscore in greet(_ name: String) suppresses the external label. Labels are part of a function's API and make multi-argument calls read like a sentence.

    Errors are explicit, but not checked exceptions

    JAVA
    String text() throws IOException {
    return Files.readString(path);
}
    KOTLIN
    fun text(): String =
    path.readText()
    SWIFT
    func text() throws -> String {
    try String(contentsOf: url)
}
let value = try text()

    Swift marks potentially failing calls with try; callers use do/catch, try?, or try!. A function may declare throws, but the compiler does not require a specific checked error type.

    Control flow and pattern matching

    Swift's switch is exhaustive and powerful. It is closer to Kotlin's exhaustive when than to Java's traditional statement switch.

    Classify an HTTP response code
    JAVA
    String category;
if (code >= 200 && code < 300) {
    category = "success";
} else if (code == 404) {
    category = "missing";
} else { category = "other"; }
    KOTLIN
    val category = when (code) {
    in 200..299 -> "success"
    404 -> "missing"
    else -> "other"
}
    SWIFT
    let category: String
switch code {
case 200...299: category = "success"
case 404: category = "missing"
default: category = "other"
}

    Control-flow tools worth learning

    • if case and guard case match enum cases or optional patterns.
    • for case let .success(value) in results filters and unwraps in one loop.
    • where adds a condition to case or a for loop.
    • defer schedules cleanup when the current scope exits, similar to a small, local finally.
    • fallthrough is explicit and uncommon; cases do not fall through by default.

    Protocols and extensions

    A Swift protocol resembles a Java interface or Kotlin interface, but extensions can add default behavior to a protocol or methods to an existing type without inheritance.

    A capability with a default implementation
    JAVA
    interface Named {
    String name();
    default String label() {
        return "@" + name();
    }
}
    KOTLIN
    interface Named {
    val name: String
    fun label() = "@$name"
}
    SWIFT
    protocol Named {
    var name: String { get }
}
extension Named {
    var label: String { "@\(name)" }
}

    Protocol vocabulary

    SyntaxPurpose
    struct User: NamedConform a type to a protocol
    any NamedStore an existential: an unknown conforming type
    some NamedReturn one specific hidden conforming type; preserves type identity
    Named & SendableProtocol composition (often written with a generic constraint instead)
    protocol P: AnyObjectRestrict conformance to classes; enables weak references
    Extension dispatch has nuance. A member declared only in a protocol extension is statically dispatched when accessed through a protocol-typed value. Put requirements in the protocol when polymorphic customization matters.

    Generics and associated types

    Swift generics are reified enough for strong optimization and expressive constraints. Protocols with associated types describe families of related types, much like Kotlin generic interfaces.

    A generic clamp operation
    JAVA
    static <T extends Comparable<T>> T clamp(
    T value, T low, T high) {
  return value.compareTo(low) < 0 ? low :
         value.compareTo(high) > 0 ? high : value;
}
    KOTLIN
    fun <T : Comparable<T>> clamp(
    value: T, low: T, high: T
) = value.coerceIn(low, high)
    SWIFT
    func clamp<T: Comparable>(
    _ value: T, low: T, high: T
) -> T {
    min(max(value, low), high)
}

    Associated types

    SWIFT
    protocol Cache {
    associatedtype Key: Hashable
    associatedtype Value
    subscript(key: Key) -> Value? { get set }
}

func preload<C: Cache>(_ cache: inout C, key: C.Key, value: C.Value) {
    cache[key] = value
}

    Because each conformer chooses its own Key and Value, any Cache cannot generally be used as freely as any Named. Prefer generic parameters or type erasure when the associated types matter.

    Value semantics and copy-on-write

    Value semantics is one of Swift's central design tools. Assigning a struct means values evolve independently; standard collections avoid eager deep copies with copy-on-write (CoW) storage.

    Independent values after assignment
    JAVA
    var a = new ArrayList<Integer>();
a.add(1);
var b = a;       // same object
b.add(2);
// a is [1, 2]
    KOTLIN
    val a = mutableListOf(1)
val b = a        // same object
b += 2
// a is [1, 2]
    SWIFT
    var a = [1]
var b = a        // value copy; storage can be shared
b.append(2)
// a is [1]

    Design implications

    • A struct method that changes self or a stored property must be marked mutating.
    • You cannot mutate a value held by let, even if one of its properties is declared var.
    • inout temporarily grants exclusive write access: increment(&score). Do not let an inout reference escape.
    • CoW means Array, Dictionary, Set, and String often copy cheaply until one owner mutates.
    Value types are not automatically “deep immutable.” A struct can contain a class reference; copying the struct then copies the reference. Choose contained types intentionally.

    ARC and object lifetime

    Swift uses Automatic Reference Counting (ARC) for class instances. Unlike JVM garbage collection, reference-count changes usually determine release immediately when the last strong reference disappears.

    Avoid a closure retain cycle
    JAVA
    class Controller {
  void start() {
    service.fetch(result -> render(result));
  }
}
    KOTLIN
    class Controller {
  fun start() {
    service.fetch { render(it) }
  }
}
    SWIFT
    final class Controller {
    func start() {
        service.fetch { [weak self] result in
            self?.render(result)
        }
    }
}

    Ownership rules

    ReferenceEffectUse
    Strong (default)Keeps object aliveNormal ownership
    weakDoes not keep object alive; always optionalBack references, delegates, callbacks that may outlive owner
    unownedDoes not keep object alive; non-optionalOnly when referenced object is guaranteed to outlive the reference
    ARC cannot collect cycles. If object A strongly owns B and B strongly owns A (directly or via closures, delegates, timers, tasks, or publishers), both leak. Break the ownership relationship deliberately, usually with weak.

    Practical debugging

    Use Xcode's Memory Graph Debugger to inspect cycles and Instruments (Leaks, Allocations) to measure behavior. Do not add weak self blindly: it can silently make required work disappear. First decide who should own the task and for how long.

    Async await and structured concurrency

    Swift concurrency expresses asynchronous work in the function type: async, throws, and actor isolation. It is closest to Kotlin coroutines, but its isolation model is built into the type system.

    Load independent values concurrently
    JAVA
    var user = userFuture.get();
var posts = postsFuture.get();
    KOTLIN
    coroutineScope {
  val user = async { loadUser() }
  val posts = async { loadPosts() }
  Screen(user.await(), posts.await())
}
    SWIFT
    async let user = loadUser()
async let posts = loadPosts()
return try await Screen(
    user: user, posts: posts
)

    Rules that prevent bugs

    • Prefer a child Task, async let, or a task group over unstructured Task { } when the work has a clear parent.
    • Cancellation is cooperative. Check Task.isCancelled or call try Task.checkCancellation() in long-running work.
    • An await is a suspension point, not necessarily a thread switch.
    • Never block an async task with semaphores or synchronous waiting for another async task.
    • Use withThrowingTaskGroup for a dynamic number of child tasks.

    Actors, Sendable, and MainActor

    Actors protect mutable state through isolation. Swift 6's strict concurrency checking asks whether values crossing concurrency domains are safe to send.

    A concurrency-safe counter
    JAVA
    class Counter {
  private int value;
  synchronized void increment() { value++; }
  synchronized int get() { return value; }
}
    KOTLIN
    class Counter {
  private var value = 0
  @Synchronized fun increment() { value++ }
  @Synchronized fun get() = value
}
    SWIFT
    actor Counter {
    private var value = 0
    func increment() { value += 1 }
    func get() -> Int { value }
}

await counter.increment()

    Key concepts

    ConceptMeaning
    actorReference type whose mutable state is isolated; calls from outside commonly require await
    @MainActorIsolates UI-facing code to the main actor; use for observable UI models and UI mutations
    SendableMarker protocol for values safely transferable across concurrency domains; value types with Sendable fields often qualify naturally
    ReentrancyAn actor can process other work while one method is suspended at await; re-check assumptions after an await
    An actor is not a transaction. Do the state update that must be atomic before an await, or validate state again after resuming. Actor isolation prevents data races; it does not make multi-step logic automatically atomic.

    SwiftUI mental model

    SwiftUI views are lightweight descriptions of UI, not long-lived Android View objects. When state changes, SwiftUI evaluates view descriptions again and reconciles the rendered interface.

    A minimal app
    JAVA / ANDROID
    public class MainActivity extends Activity {
  protected void onCreate(Bundle state) {
    super.onCreate(state);
    setContentView(R.layout.main);
  }
}
    KOTLIN / COMPOSE
    class MainActivity : ComponentActivity() {
  override fun onCreate(state: Bundle?) {
    super.onCreate(state)
    setContent { Text("Hello") }
  }
}
    SWIFT / SWIFTUI
    import SwiftUI

@main
struct SampleApp: App {
    var body: some Scene {
        WindowGroup { Text("Hello") }
    }
}

    Read this declaration literally

    body returns some View: one concrete, compiler-known view type hidden behind an opaque return type. Its result is a value description. Avoid storing a view to “update it”; store the model state, then let body derive the view hierarchy.

    Compose comparison: SwiftUI and Compose share declarative state-driven rendering. SwiftUI uses property wrappers such as @State and @Binding, and view identity is especially important for preserving state and animations.

    SwiftUI state and data flow

    The most important SwiftUI skill is placing state at the right owner and passing only the access a child needs.

    ToolUse it whenCompose analogy
    @StateThis view owns simple local value stateremember { mutableStateOf(...) }
    @BindingA child edits state owned by a parentState hoisting: value + callback
    @Observable + @StateThis view owns a reference model using ObservationState holder / observable model
    @EnvironmentRead shared contextual dependencies or valuesCompositionLocal
    @AppStoragePersist a small preference in UserDefaultsDataStore preference, conceptually
    Parent owns state; child receives a binding
    JAVA
    // Traditional UI: parent receives listener
toggle.setOnCheckedChangeListener(
  (button, checked) -> enabled = checked
);
    KOTLIN / COMPOSE
    @Composable fun Toggle(
  enabled: Boolean, onChange: (Boolean) -> Unit
) { Switch(enabled, onCheckedChange = onChange) }
    SWIFTUI
    struct SettingsView: View {
    @State private var enabled = false
    var body: some View {
        ToggleRow(enabled: $enabled)
    }
}
struct ToggleRow: View {
    @Binding var enabled: Bool
    var body: some View { Toggle("Enabled", isOn: $enabled) }
}
    Do not create model objects in body. A view's body can run often. Store owned observable reference state with an appropriate property wrapper so its lifetime follows view identity rather than one body evaluation.

    Accessibility checklist

    • Use native controls before custom gestures; they provide labels, roles, focus, and behavior.
    • Add accessibilityLabel when visible text is absent or unclear; do not repeat obvious labels.
    • Support Dynamic Type: prefer flexible stacks, wrapping, and scalable text over fixed height and tiny fonts.
    • Do not convey state only through color; test contrast and VoiceOver reading order.
    • Respect Reduce Motion for nonessential, attention-heavy animation.
    • Test with Xcode Previews across sizes, color schemes, accessibility text sizes, and device types.

    Networking, data, and persistence

    Apple frameworks favor explicit types and async APIs. A common SwiftUI architecture keeps networking and persistence outside views, exposes a small observable model, and renders state.

    Decode JSON with an async request
    JAVA
    var request = HttpRequest.newBuilder(uri).build();
var body = client.send(request,
    BodyHandlers.ofString()).body();
var player = mapper.readValue(body, Player.class);
    KOTLIN
    val body = client.get(url).bodyAsText()
val player = json.decodeFromString<Player>(body)
    SWIFT
    struct Player: Decodable, Identifiable {
    let id: UUID
    let name: String
}
let (data, _) = try await URLSession.shared.data(from: url)
let player = try JSONDecoder().decode(Player.self, from: data)

    Performance and profiling

    Swift can be very fast, but elegant code can still allocate, copy, hash, bridge, or hop between actors more than intended. Measure on a realistic device and workload before optimizing.

    High-value rules

    AreaGuidance
    CollectionsReserve capacity for known growth; avoid repeatedly rebuilding large arrays or strings in hot paths.
    Value typesCoW makes passing collections cheap until mutation; avoid accidental uniqueness breaks and unnecessary mutation after copying.
    AbstractionUse generics for hot, type-preserving paths; assess any Protocol only if profiling identifies existential overhead.
    SwiftUIKeep body cheap and deterministic; use stable identity; avoid expensive work in view evaluation; move work to models/tasks.
    ConcurrencyDo not create a task per tiny operation; batch work and avoid unnecessary actor hops.
    BridgingRepeated conversions between Swift collections/strings and Foundation or Objective-C types can cost time and allocations.

    Instruments first

    Time Profiler: CPU hotspots→Allocations: churn→Leaks: cycles→SwiftUI: render behavior→Points of Interest: signposts

    Properties, initialization, and access control

    Swift makes initialization and stored-property ownership explicit. This avoids partially initialized objects and helps preserve value invariants.

    A computed property with controlled mutation
    JAVA
    final class Temperature {
  private double celsius;
  double fahrenheit() {
    return celsius * 9 / 5 + 32;
  }
  void setCelsius(double value) {
    if (value < -273.15) throw new IllegalArgumentException();
    celsius = value;
  }
}
    KOTLIN
    class Temperature(celsius: Double) {
  var celsius = celsius
    set(value) {
      require(value >= -273.15)
      field = value
    }
  val fahrenheit get() = celsius * 9 / 5 + 32
}
    SWIFT
    struct Temperature {
    var celsius: Double {
        didSet { precondition(celsius >= -273.15) }
    }
    var fahrenheit: Double { celsius * 9 / 5 + 32 }
    init(celsius: Double) {
        precondition(celsius >= -273.15)
        self.celsius = celsius
    }
}

    Property vocabulary

    FeatureUse
    Stored propertyPersists a value in each instance: var score: Int.
    Computed propertyDerives a value with get and optionally set; it has no independent storage.
    ObserverswillSet and didSet react after initialization; use sparingly for local consistency, not hidden I/O.
    lazy varDefers creation until first access. It must be a variable because first access mutates storage.
    Property wrapperPackages storage behavior behind @Wrapper; SwiftUI's @State is the most visible example.

    Access control

    Swift access levels are open, public, package, internal (default), fileprivate, and private. Use public for an API visible to importing modules; use open only when external modules may subclass or override a class API. Prefer the narrowest level that supports the API.

    SwiftUI architecture and dependency injection

    SwiftUI does not require a named architecture, but successful apps consistently separate rendering, screen state, domain rules, and I/O. Keep the view declarative and make effects explicit.

    A practical feature shape

    View: renders state and sends intent→Model: coordinates feature state→Service protocol: defines I/O→Implementation: URLSession, persistence, platform API
    Inject an interface/protocol instead of creating a global service
    JAVA
    final class ScoreController {
  private final ScoreService service;
  ScoreController(ScoreService service) {
    this.service = service;
  }
}
    KOTLIN
    class ScoreModel(
  private val service: ScoreService
)
    SWIFT
    protocol ScoreService: Sendable {
    func scores() async throws -> [Int]
}
@Observable @MainActor
final class ScoreModel {
    private let service: any ScoreService
    init(service: any ScoreService) {
        self.service = service
    }
}

    State design rules

    • Represent screen states explicitly: loading, content, empty, and error should be visible in the model instead of inferred from several unrelated booleans.
    • Make the view's inputs obvious. Use initializer injection for required dependencies; use environment injection for shared app-level services.
    • Keep business logic out of body. Views can format and compose; services and models perform I/O and decisions.
    • Use a fake conformer in previews and tests. A preview that reaches the network is neither reliable nor fast.
    • Do not force every feature into a giant global store. State should live at the lowest owner that needs to coordinate it.

    SwiftUI connects work to view identity through task modifiers and groups windows into scenes. Treat lifecycle events as signals to reconcile state, not as a substitute for a durable data model.

    Privacy, localization, and release readiness

    An app that works in a simulator is not yet a production Apple app. Privacy declarations, permission text, localization, accessibility, diagnostics, and store metadata are product requirements.

    Privacy rules

    • Request the minimum capability and explain the user benefit at the moment it is needed.
    • Provide the corresponding usage-description strings for protected resources such as camera, microphone, photos, location, Bluetooth, contacts, and health data; the system can terminate an app missing a required description.
    • Store secrets in Keychain and apply platform privacy manifests, data-use declarations, and third-party SDK requirements accurately.
    • Use Logger's privacy interpolation. Treat dynamic strings as private by default unless they are safe to expose.

    Before TestFlight and App Store submission

    • Test real-device permissions, offline recovery, and account deletion/sign-out flows.
    • Check every supported language, right-to-left behavior where relevant, Dynamic Type, VoiceOver, and contrast.

    Java and Kotlin migration guide

    Do not transliterate classes and callbacks line by line. Port behavior while adopting Swift's value semantics, optionals, protocol-oriented boundaries, and structured concurrency.

    Recommended migration sequence

    1. Extract domain rules. Port pure models and calculations first. Make simple data structs and use enums for finite state.
    2. Replace null contracts. Turn nullable Java/Kotlin values into T?; decide at each use site whether absence is valid, defaultable, or an error.
    3. Redesign service boundaries. Replace interface-heavy inheritance trees with focused protocols, generic algorithms, and injected dependencies.
    4. Adopt async APIs. Convert callback/future/coroutine boundaries to async throws; introduce actors around shared mutable services.
    5. Build state-driven UI. Model screen state, then render it in SwiftUI. Do not keep a mutable tree of views.
    6. Validate on platform. Test Dynamic Type, VoiceOver, lifecycle events, offline behavior, background constraints, and real devices.

    Translation table

    Java / Kotlin habitSwift / SwiftUI direction
    DTO class with gettersImmutable struct with stored let properties; conform to Codable if appropriate
    Nullable field checked laterOptional; unwrap at the boundary where you can make a useful decision
    Abstract base classProtocol plus extension; use a class only for shared identity/lifecycle
    Singleton with locksDependency-injected actor when mutable shared state is necessary
    Coroutine scope / executorStructured child tasks, task groups, actor isolation, @MainActor
    RecyclerView / Compose lazy listList, ForEach, or lazy stack with stable Identifiable identity
    Mutable UI widget referencesState plus declarative view description; mutate state, not the view
    A reliable first project: build a small offline-first list-and-detail app. It exercises models, optionals, collections, navigation, state, persistence, async loading, errors, accessibility, and testing without hiding the language behind a large architecture.

    Final checklist

    • Prefer struct until shared identity is a real requirement.
    • Model absence and failure explicitly with optionals and errors.
    • Use protocol boundaries where testing or variation requires them; do not create interfaces by reflex.
    • Make concurrency isolation part of the design, not a compiler-warning cleanup task.
    • Keep SwiftUI views declarative and state-driven.
    • Profile before optimizing and test on the Apple platforms you ship.
